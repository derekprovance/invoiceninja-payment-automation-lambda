## Project Overview

**invoiceninja-payment-automation-lambda** is an AWS Lambda function that listens to Amazon SES email events, parses Venmo payment notifications, and automatically records payments in Invoice Ninja. It handles client identification, invoice matching, and payment allocation across multiple invoices.

## Architecture

### Entry Point & Execution Flow

```
index.ts (Lambda handler)
  └─ EmailEventHandlingService.handleEmailEvent()
       ├─ Validates SES event structure
       └─ PaymentFactory.createPayment()
            └─ VenmoPayment (parses email for client name & amount)
  └─ PaymentProcessingService.processPayment()
       ├─ getClient() — two-pass name matching (exact name, then contacts)
       ├─ listInvoices() — retrieves unpaid invoices for client
       ├─ allocatePayment() — payment allocation algorithm
       ├─ createPayment() — posts payment to Invoice Ninja API
       └─ createCredit() — explicitly creates credit for overpayment surplus
```

### Payment Allocation Algorithm

The `allocatePayment()` method in `PaymentProcessingService` uses a **two-pass strategy**:

1. **Pass 1 — Exact Match**: If exactly one invoice has a balance equal to the payment amount (within floating-point epsilon of 0.001), pay only that invoice. This handles cases where the customer is settling a specific bill.

2. **Pass 2 — Oldest-First Fallback**: If no unique exact match exists, sort invoices by `date` (YYYY-MM-DD, ascending), with `created_at` (Unix timestamp) as a tiebreaker, and greedily apply the payment amount. This ensures customers settle older debts first.

**Edge Case**: Multiple exact matches (e.g., two $10 invoices, $10 payment) fall through to oldest-first sorting—ambiguous cases don't guess.

**Invoice Data**: Both `date` and `created_at` fields are optional in the `InvoiceNinjaInvoice` interface; the sorting gracefully handles missing data.

### Client Matching (Two-Pass)

**Pass 1**: Searches for exact name match (case-insensitive) in the `name` field of returned clients.

**Pass 2**: If no single match, fetches full client details (via `getClientById()`) and searches contact records for matches on `first_name` + `last_name` (case-insensitive).

**Error Handling**: Throws `UnhandledScenarioError` if multiple clients match (ambiguous). Returns `no_client` status if no match.

### Data & Status

- **Invoice Status**: Status ID `'4'` means "Paid" in Invoice Ninja.
- **Currency Precision**: Epsilon threshold is `0.001` to handle floating-point rounding in payment totals.
- **Repository Pattern**: All Invoice Ninja API calls go through `InvoiceNinjaRepository`, which wraps `axios` with error handling and request logging.
- **Credit Creation**: When a payment amount exceeds the sum of allocated invoices (overpayment), a credit is explicitly created via `POST /credits` + `mark_sent=true` so the surplus is immediately available as a client credit for future invoices. Invoice Ninja does not auto-create credits when a payment is posted.

## Commands

### Building & Deployment

- `npm run compile` — Bundles TypeScript, minifies with esbuild, outputs to `dist/index.zip` for Lambda upload.
- `npm run build` — Alias for compile (also available as just `npm run build`).
- `npm run type-check` — Runs TypeScript compiler without emitting files (catch type errors before compile).
- `npm run format` — Formats all `.ts` files with Prettier.

### Testing

- `npm run test` — Runs all unit tests in the `src/**/*.test.ts` pattern.
- `npm run test:watch` — Runs unit tests in watch mode (re-runs on file changes).
- `npm run test:integration` — Runs integration tests (requires Docker; see below).
- `npm run test:all` — Runs both unit and integration test suites in sequence.

**Unit Test Files**: Located as `*.test.ts` siblings to source files.

### Integration Testing (Docker)

Integration tests run against a live Invoice Ninja instance in Docker Compose.

#### Quick Start
```bash
make integration:test
```
This target automatically starts Docker, initializes Invoice Ninja, runs tests, and cleans up on completion (or failure).

#### Manual Steps
```bash
make integration:start    # Start Docker + init Invoice Ninja
npm run test:integration  # Run tests
make integration:stop     # Stop Docker
make integration:clean    # Remove Docker volumes (if needed for cleanup)
```

#### Setup Notes
- `.env.integration` must exist with `IN_APP_KEY=base64:...` (auto-generated by `make integration:key`).
- Docker Compose spins up: MariaDB, Redis, Invoice Ninja, nginx.
- Tests verify: exact client matching, contact matching, multi-invoice allocation, partial/over payments, invoice status updates.
- Each test creates and cleans up its own data via `InvoiceNinjaTestClient`.

#### Troubleshooting
- **InvoiceNinja won't start**: Run `make integration:key` to generate the APP_KEY.
- **Tests fail with stale data**: Run `make integration:clean` to remove Docker volumes and recreate fresh.
- **ECONNREFUSED or timeout**: Docker services may be slow. Check `docker compose ps` and container logs: `docker compose logs invoiceninja`.

## Configuration

### Environment Variables

Required (must be set):
- `BASE_URL` — Invoice Ninja base URL (e.g., `http://localhost:8000` for self-hosted or cloud domain).
- `TOKEN` — Invoice Ninja API token.
- `VENMO_EMAIL` — Email address from Venmo notifications (default: `venmo@venmo.com`).

Optional:
- `VENMO_PAYMENT_GATEWAY_ID` — Gateway ID in Invoice Ninja for Venmo (default: `25`).
- `LOG_LEVEL` — Logging level (default: `info`; use `debug` for detailed output).

### Integration Testing Environment

- `.env.integration` — Auto-generated by Makefile if missing. Contains `IN_APP_KEY` and other test-specific variables.
- `docker/integration.env` — Generated by `init-invoiceninja.sh` after Invoice Ninja starts; contains auth token for test client.

## Testing Strategy

### Unit Tests
- Mock `IInvoiceRepository` interface; test `PaymentProcessingService` logic in isolation.
- Verify: client matching, name normalization, exact-match and oldest-first allocation, float tolerance, error handling.
- Run with `npm run test` (~30 tests).

### Integration Tests
- Run against a real Invoice Ninja instance via Docker.
- Verify: end-to-end payment processing, SES → Lambda → Invoice Ninja flow.
- Test data created and cleaned up per test (`afterEach` calls `testClient.cleanupAll()`).
- Run with `npm run test:integration` (~12 tests).

### Test Configuration (vitest)
- **Projects**: Two named projects: `unit` and `integration`.
- **Integration Setup**: `src/tests/integration/setup/envSetup.ts` runs before tests to load environment variables from `docker/integration.env`.
- **Pool**: Integration tests use `pool: 'forks'` to ensure each worker is a fresh process (required so `setupFiles` executes before config module loads).
- **Timeout**: Integration tests have 30-second timeout.

## Test Scenarios (Do Not Deviate)

**CRITICAL**: These test scenarios define the contract for this system. Changes to the codebase must preserve all passing tests. New features must not break any scenario listed below.

### Unit Test Scenarios (PaymentProcessingService.test.ts)

#### Core Status Tests
1. **No client found** — Payment for unknown name → `no_client` status returned
2. **No invoices found** — Payment for client with no invoices → `no_invoice` status returned
3. **Basic success** — $50 payment to client with $50 invoice → `success` status returned

#### Float Tolerance
4. **Floating-point precision** — $10.1 + $20.2 = $30.3 (via two invoices) paid by $30.3 payment → `success` with float tolerance epsilon of 0.001; no credit created (surplus ~2.8e-14 below epsilon)

#### Client Name Matching (Two-Pass)

**Pass 1 — Exact Name Match**:
5. **Exact match with superset** — Client names ["John Smith", "John Smith Jr."] with payment for "John Smith" → matches exactly "John Smith" (not "Jr." variant)
6. **Case-insensitive match** — Client name "JOHN SMITH" with payment for "John Smith" → matches (normalized comparison)
7. **Name mismatch** — Client name "Johnson Smith" with payment for "John Smith" → `no_client` (no partial matches)

**Pass 2 — Contact Name Fallback**:
8. **Contact match** — Client name "Smith Family" with contact "John Smith" and payment for "John Smith" → finds client via contact
9. **Multiple client ambiguity (name)** — Two clients with same name → throws `UnhandledScenarioError`
10. **Multiple client ambiguity (contacts)** — Two clients with same contact name → throws `UnhandledScenarioError`

#### Payment Allocation Scenarios

**Scenario 1: Single Invoice, Partial Payment**:
11. **Partial payment to single invoice** — $20 payment to $30 invoice → $20 allocated to that invoice

**Scenario 2: Multiple Invoices, Complete Allocation**:
12. **Two invoices, $10 + $30, $20 payment** → $10 fully paid (oldest), $10 partial on $30 (newest) → both invoices included in allocation

**Scenario 3: Overpayment**:
13. **Overpayment** — $20 payment to $10 invoice → $10 allocated to invoice, full $20 sent to Invoice Ninja API, surplus ($10) explicitly created as a credit. The test verifies `createCredit` is called with the surplus amount.

**Scenario 4: Exact Match Override**:
14. **Exact match takes precedence** — Invoices [$7 oldest, $10 newer] with $10 payment → pays only the $10 invoice (not oldest), ignoring the older $7

### Integration Test Scenarios (payment.integration.test.ts)

#### Client & Invoice Matching
1. **Exact client name match** — Client "Alice Smith" with $150 invoice, payment for "Alice Smith" $150 → invoice marked paid (status_id = '4')
2. **Contact name match** — Client "Smith Family" with contact "John Smith", payment for "John Smith" → payment succeeds (contact matching works)

#### Invoice Allocation
3. **Multi-invoice allocation** — Client with [$20, $30] invoices, $50 payment → both invoices marked paid
4. **Partial payment** — Client with $30 invoice, $20 payment → invoice stays open (not paid), $20 recorded in payments
5. **Overpayment** — Client with $10 invoice, $20 payment → invoice marked paid, full $20 recorded in payment record, surplus ($10) created as credit on client

#### Allocation Ordering
6. **Three-invoice oldest-first** — Client with [$10, $25, $40] invoices (created oldest→newest), $35 payment → $10 + $25 paid (oldest first), $40 untouched
7. **Older but larger invoice prioritized** — Client with [$30 old, $10 new] invoices, $15 payment → $15 applied to $30 (oldest, partial), $10 untouched (newer)

#### Edge Cases
8. **Float tolerance** — Client with [$10.1, $20.2] invoices, $30.3 payment → both invoices marked paid, no spurious credit (0)
9. **Paid invoice excluded** — Client with [$50 previously paid, $30 unpaid], $30 payment → payment goes to $30 unpaid invoice (not $50 which is already paid)

#### Exact Match Behavior
10. **Exact match ignores age** — Client with [$10 oldest, $7.55 newer], $7.55 payment → $7.55 invoice marked paid (exact match takes precedence over oldest-first), $10 untouched

#### Error Handling
11. **No matching client** — Payment for "Ghost Person" → `no_client` status returned
12. **Client with no invoices** — Client "Carol White" exists but has no invoices, payment attempted → `no_invoice` status returned

### Expected Status Return Values

The Lambda handler must return one of these status values in all cases:
- `success` — Payment was allocated and recorded in Invoice Ninja
- `no_client` — No client found matching the payment name
- `no_invoice` — Client found but has no unpaid invoices
- **Errors**: Throws `UnhandledScenarioError` when client/contact names are ambiguous (multiple exact matches)
